1. Executive Summary
Platform Identity
RSN (Pods of Reason) is a Connection Operating System — not social media, not a community feed, not just an
event platform. It encodes a social contract enabling intentional, trust-based, non-transactional human connection at
scale.
Core Principle
"If a feature increases noise, performance pressure, or selling behaviour, it is wrong by default."

Milestone Members Timeline

First 500: Free for life (cultural anchors)
Entry: Events are always free
Conversion: After 3rd participation → Invite 3-5 OR pay OR both

meeting_frequency TEXT,

recording_enabled BOOLEAN DEFAULT false;
-- New: Pod Directors Table
CREATE TABLE pod_directors (
id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
pod_id UUID REFERENCES pods(id) ON DELETE CASCADE,
user_id UUID REFERENCES users(id) ON DELETE CASCADE,
assigned_by UUID REFERENCES users(id),
assigned_at TIMESTAMPTZ DEFAULT NOW(),
UNIQUE(pod_id, user_id)
);
-- New: Pod Join Requests Table
CREATE TABLE pod_join_requests (
id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
pod_id UUID REFERENCES pods(id) ON DELETE CASCADE,

status TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected')),
message TEXT,
reviewed_by UUID REFERENCES users(id),
created_at TIMESTAMPTZ DEFAULT NOW(),
reviewed_at TIMESTAMPTZ,
UNIQUE(pod_id, user_id)
);
-- New: Pod Meetings Table
CREATE TABLE pod_meetings (
id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
pod_id UUID REFERENCES pods(id) ON DELETE CASCADE,
title TEXT NOT NULL,
description TEXT,
scheduled_at TIMESTAMPTZ NOT NULL,
duration_minutes INTEGER DEFAULT 60,
meeting_url TEXT,
meeting_type TEXT DEFAULT 'zoom' CHECK (meeting_type IN ('zoom', 'google_meet', 'other')),
created_by UUID REFERENCES users(id),

);
-- New: Pod Chat Messages Table
CREATE TABLE pod_chat_messages (
id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

deleted_at TIMESTAMPTZ
);
-- New: Violation Reports Table
CREATE TABLE violation_reports (
id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

is_anonymous BOOLEAN DEFAULT false,
violation_type TEXT NOT NULL,

max_pods_owned INTEGER DEFAULT 1,
max_pod_size INTEGER DEFAULT 6,
max_directors_per_pod INTEGER DEFAULT 0,
can_create_orchestra BOOLEAN DEFAULT false,
can_enable_recording BOOLEAN DEFAULT false,
is_lifetime_free BOOLEAN DEFAULT false, -- First 500
created_at TIMESTAMPTZ DEFAULT NOW(),
updated_at TIMESTAMPTZ DEFAULT NOW()

-- New: Activity Log Table (for Reason Score calculation)

activity_type TEXT NOT NULL,
activity_data JSONB,

);

Decision 3: CDN Strategy
Option Pros Cons

Vercel Edge Integrated with deployment Tied to Vercel hosting

Risk Assessment:
Cache invalidation issues: LOW (hashed filenames)
Regional availability: LOW (Cloudflare is global)
Decision 4: Caching Layer
Option Pros Cons
No Cache (Supabase direct) Simple, always fresh Higher latency, more DB load
Supabase Edge Functions cache Built-in, simple Limited control, cold starts
Upstash Redis Fast, serverless Redis Additional service ($10-50/mo)
In-memory (per request) Free, simple No persistence between requests
Recommendation: Start with in-memory request caching, add Upstash Redis for rate limiting and sessions when
needed.
Use Case Caching Strategy
Rate limiting Upstash Redis (required at scale)
Session data Supabase Auth (built-in)
User entitlements In-memory + short TTL
Hot data (active pods) In-memory per request
Search results No cache (always fresh)
Scale Threshold:
Users Strategy
< 1,000 In-memory only
1,000-5,000 Add Upstash for rate limiting
> 5,000 Full Redis caching layer
Cost Implications:
Users Upstash Cost
1,000 $10/mo
5,000 $25/mo
10,000 $50/mo
Implementation Effort: 2 dev days (rate limiting), 5 dev days (full caching)
Risk Assessment:
Cache stampede: MEDIUM (implement locks)

Redis unavailability: LOW (graceful degradation)

6.5 Security & API Decisions
Decision 9: Authentication Token Strategy
Option Pros Cons
Supabase Auth (JWT) Built-in, secure, handles refresh Tied to Supabase
Custom JWT Full control Security responsibility, complexity
Session cookies Simple, secure Not suitable for mobile/API
Recommendation: Use Supabase Auth with JWT, leverage built-in refresh token handling.
Token Type Lifetime Storage
Access token 1 hour Memory
Refresh token 7 days Secure cookie
Magic link 1 hour Email only
Security Measures:
 HttpOnly cookies for refresh tokens
 CSRF protection on state-changing requests
 Rate limiting on auth endpoints (5/min)
 Session invalidation on password change
 Max 5 concurrent sessions per user
Implementation Effort: Already implemented via Supabase Auth
Risk Assessment:
Token theft: LOW (HttpOnly, short lifetime)
Refresh token abuse: LOW (rotation on use)
Session fixation: LOW (Supabase handles)

Risk Severity Impact
Resend rate limits HIGH Failed invite/notification emails
Spam classification HIGH Invites going to spam
Bounce handling MEDIUM Invalid emails degrading reputation
Mitigation Strategies:
1. Implement email queuing with retry logic
2. Set up dedicated sending domain with proper DNS (SPF, DKIM, DMARC)
3. Monitor bounce rates and auto-disable bad addresses
4. Implement email verification at signup
5. Consider dedicated IP as volume grows
6. Rate limit: max 100 emails/user/day
6.4 Invite System Abuse
Risk Severity Impact
Spam invites HIGH Platform reputation damage
Fake account farms HIGH Inflate numbers, degrade trust
Invite manipulation MEDIUM Gaming the system for benefits
Mitigation Strategies:
1. Daily invite caps (start: 3/day, scale with trust)
2. Invite cooldown (1 hour between invites to same domain)
3. LinkedIn verification required (blocks throwaway accounts)
4. IP-based rate limiting
5. Machine learning fraud detection (Phase 2)
6. Manual review queue for suspicious patterns
6.5 Pod Chat Architecture
Risk Severity Impact
Message storage growth HIGH 10K users × 10 pods × 100 messages/week = 10M rows/week
Real-time sync HIGH Supabase Realtime limits
Search functionality MEDIUM Full-text search at scale
Mitigation Strategies:
1. Implement message pagination (load last 50, lazy load older)
2. Archive messages > 90 days to separate table
3. Use Supabase Realtime for small pods only (< 50 members)
4. Larger pods: polling every 5 seconds
5. Implement full-text search using pg_trgm extension
6. Consider message size limits (5KB max)

Reason Score Algorithm
Purpose: Measure member contribution and trustworthiness without creating popularity dynamics.
Scoring Components (0.0 - 1.0 scale)
Factor Weight Calculation
Profile Completeness 10% Fields filled / Total fields
Event Participation 20% Events attended / Events available (capped at 1.0)
Pod Engagement 25% Active days in pods / Total membership days
Invite Quality 20% (Invites that became active members) / Total invites
Governance Record 15% 1.0 - (Strikes * 0.33)
Tenure Bonus 10% min(1.0, Months as member / 12)
Calculation Formula
reason_score = (
(profile_completeness * 0.10) +
(event_participation * 0.20) +
(pod_engagement * 0.25) +
(invite_quality * 0.20) +
(governance_record * 0.15) +
(tenure_bonus * 0.10)
)
Display Rules
Never show numeric score to users
Use qualitative indicators: "Trusted", "Active", "New"
Admin can see exact scores
Score updates daily via scheduled function
Score Thresholds
Score Display Benefits
0.0 - 0.3 New Member Standard access
0.3 - 0.6 Active Increased invite cap
0.6 - 0.8 Trusted Director eligibility
0.8 - 1.0 Trusted Voice Badge, priority matching

16. Appendix
A. Technology Decision Records
ADR-001: Supabase as Backend
Context: Need managed backend with auth, database, real-time
Decision: Use Supabase (PostgreSQL, Auth, Edge Functions, Realtime)
Consequences: Vendor lock-in, but rapid development and built-in scaling
ADR-002: React + Vite for Frontend
Context: Need modern, fast frontend with good DX
Decision: React 18 with Vite build tool
Consequences: Large ecosystem, easy hiring, fast development
ADR-003: shadcn/ui for Components
Context: Need accessible, customizable component library
Decision: Use shadcn/ui (copy-paste components)
Consequences: Full control, no dependency lock-in, Tailwind required
ADR-004: Stripe for Payments
Context: Need reliable subscription billing
Decision: Stripe with Customer Portal
Consequences: Industry standard, easy integration, higher fees
